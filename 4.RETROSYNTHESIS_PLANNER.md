# RetroSynthesis Planner (Tech README)

# IMP


## Tech stack
- Frontend: Next.js (App Router) + React + TypeScript
- Backend: Python + FastAPI
- Core planner library: `backend/retro_star/` (Retro*)
  - Uses PyTorch (`torch`)
  - Uses RDKit + rdchiral for reaction-template application
  - Uses a neural ‚Äúone-step‚Äù policy model (MLP) and (optionally) a value network

## Logic (end-to-end request flow)
1. User opens `frontend/app/retrosynthesis/page.tsx` and enters a **target SMILES**.
2. Frontend calls backend with:
   - `POST /retrosynthesis/plan`
   - body: `{ smiles: "<target smiles>" }`
3. Backend endpoint `backend/routers/retro_router.py`:
   - validates the request body
   - forwards the SMILES to `backend/retro_module.py: plan(smiles)`
4. `backend/retro_module.py`:
   - lazily creates a single `RSPlanner` instance (`_planner` cache)
   - calls `_get_planner().plan(smiles)`
   - parses the returned serialized route into structured steps (`parse_route_string`)
5. Frontend renders:
   - a success/failure banner
   - for successful multi-step plans: each step shows product, probability, and reactants.

## Technical concepts used
- Neural-guided search / planning
  - Retro* uses a neural policy to propose reaction templates and guide search.
- A* style planning abstraction (as described in Retro* paper/README)
- Lazy initialization / singleton pattern for heavy models
  - `retro_module._get_planner()` initializes RSPlanner only once per backend process.
- String serialization parsing
  - Retro* returns a `routes` string; `parse_route_string()` converts it into a list of steps.

## Algorithms (what is happening under the hood)
### 1) Retro* neural-guided A* search (planner algorithm)
The included Retro* library describes the approach as:
- **‚ÄúRetro*: Learning Retrosynthetic Planning with Neural Guided A* Search‚Äù**
- A learned policy proposes candidate one-step retrosynthesis moves.
- Search iteratively expands partial routes using the neural guidance and (optionally) a learned value function.

In this repo‚Äôs RSPlanner configuration (in `backend/retro_module.py`):
- `use_value_fn=True`
- `iterations=100`
- `expansion_topk=50`
- `gpu=-1` (CPU mode)

### 2) Applying reaction templates to generate precursors
Within Retro*‚Äôs MLP policy module:
- Top-k neural predictions select reaction templates (rules).
- `rdchiral` is used to run the text-based reaction templates (`rdchiralRunText`) to generate candidate reactants.
- Template candidates and probabilities are normalized and merged.

### 3) Route parsing (backend formatting for UI)
`retro_module.parse_route_string(route_str)`:
1. Splits the serialized `route_str` on `|` into ‚Äúreaction‚Äù chunks.
2. Each chunk is split on `>` into 3 parts:
   - `product`
   - `probability`
   - `reactants` (dot-separated SMILES strings)
3. Produces:
   - `[{ product, probability, reactants: [...] }, ...]`

## Data structures
- Frontend:
  - `PlanResult` (typed) with:
    - `success`, `smiles`, `already_buyable`
    - `routes: ReactionStep[]` where each step has `product`, `probability`, `reactants[]`
- Backend:
  - RSPlanner singleton:
    - `_planner: RSPlanner | None`
  - Route formatting:
    - list of step dicts from `parse_route_string`
  - Planner returns:
    - dict with keys like `succ`, `time`, `iter`, and serialized `routes` + cost/length.
- Retro* library internals (from the code excerpt):
  - neural policy uses tensors and `torch.topk` to pick top candidates
  - stores temporary candidate reaction/precursor mappings in dictionaries and `defaultdict(list)`

## Dataset(s) / required model assets
Retro*‚Äôs RSPlanner uses default paths inside `backend/retro_star/retro_star/`:
- Starting molecules:
  - `dataset/origin_dict.csv`
- One-step template rules:
  - `one_step_model/template_rules_1.dat`
- One-step policy checkpoint:
  - `one_step_model/saved_rollout_state_1_2048.ckpt`
- Value network checkpoint (value function):
  - `saved_models/best_epoch_final_4.pt`

If these assets are missing, planning will fail or degrade (depending on how the Retro* package handles missing files).
