# SMILES Converter Studio (Tech README)

## Tech stack
- Frontend: Next.js (App Router) + React + TypeScript
- Styling/UI: Tailwind CSS + shadcn/ui components
- Backend: Python + FastAPI (REST API)
- Chemistry engine: RDKit (SMILES/InChI parsing, property descriptors, 2D depictions)
- Data access:
  - PubChem REST API (name/formula → SMILES and compound lookup)
  - CSV + SDF local files loaded by the backend

## Logic (high-level data flow)
1. User enters a molecule input on the `frontend/app/converter/page.tsx` UI (SMILES/InChI/name/formula).
2. Frontend calls backend `POST /chemistry/analyze-enhanced` with `{ smiles: <input> }`.
3. Backend:
   - Converts input into an RDKit molecule (`smiles_to_mol`), using:
     - direct SMILES parsing
     - InChI parsing if input starts with `InChI=`
     - PubChem name/formula resolution as a fallback
   - Generates:
     - a 2D structure image (base64 PNG)
     - canonical SMILES, InChI, InChIKey, and MOL block (V2000)
   - Computes molecular descriptors (“properties”)
   - Detects functional groups and estimates a reactivity tier
   - Builds an element-by-element “Bohr model” view by joining element composition with periodic-table metadata
4. Frontend renders the results in tabbed sections (Structure, Properties, Bohr Models, Reactivity) and exposes identifier copy/download actions.

## Technical concepts used
- REST API design with pydantic schemas (FastAPI request/response models)
- Molecular informatics with RDKit:
  - SMILES/InChI parsing
  - descriptor calculation (MolWt, LogP, TPSA, HBD/HBA, rings, rotatable bonds)
  - substructure matching via SMARTS
- Fallback resolution strategy (local dataset → PubChem API)
- Server-side caching for performance:
  - periodic table cache
  - SDF molecule cache
  - SMILES dataset (pandas DataFrame) cache
- Client-side state management for multi-tab visualization and dataset browsing

## Algorithms
### 1) Input → molecule conversion
- Parse SMILES with sanitization disabled first to improve robustness.
- If parsing fails and input looks like InChI, parse via `Chem.MolFromInchi`.
- If still missing, try PubChem resolution by searching:
  - `name` namespace
  - `formula` namespace
- Finally, sanitize (best-effort), add hydrogens, and generate 2D coordinates.

### 2) Functional group / reactivity detection
- Define a dictionary of SMARTS patterns (e.g., Aldehyde, Ketone, Carboxylic Acid, Ester, Alcohol, Amine, Nitro, Thiol, Sulfide, Halogen).
- For each SMARTS pattern:
  - compile with `Chem.MolFromSmarts`
  - detect presence using `mol.HasSubstructMatch(pattern)`
- Reactivity tier is derived from the number of detected groups:
  - 0 → “Low (Inert)”
  - 1–2 → “Moderate”
  - 3–4 → “High”
  - 5+ → “Very High (Highly Reactive)”

### 3) Drug-likeness (Lipinski’s Rule of Five)
- Compute:
  - Molecular Weight (≤ 500)
  - LogP (≤ 5)
  - HBD (≤ 5)
  - HBA (≤ 10)
- Report violations and output Pass/Fail.

### 4) Bohr model enrichment
- Count atom symbols in the molecule (`mol.GetAtoms()`).
- For each element symbol:
  - look up element metadata from the periodic-table CSV cache (Bohr model images + 3D model link + electron configuration + electronegativity).

### 5) Dataset search (bioactivity / property dataset)
- The backend loads `SMILES_Big_Data_Set.csv` into a pandas DataFrame with columns:
  - `SMILES`, `pIC50`, `num_atoms`, `logP`
- Query strategies:
  - if query parses as a float pIC50 target: return molecules within ±0.5 pIC50, sorted by pIC50 desc
  - otherwise: substring match of query inside the `SMILES` column

## Data structures
- Backend caches (in-memory):
  - `dict[str, dict]` periodic table rows keyed by `symbol`
  - `dict[str, Chem.Mol]` SDF molecule cache keyed by canonical SMILES
  - `dict[str, list[str]]` molecule names keyed by canonical SMILES
  - `pandas.DataFrame` for the SMILES dataset
- React state in the converter UI:
  - `result`: typed response object containing properties/reactivity/Bohr models
  - `datasetResults`: array of dataset matches (smiles/pIC50/num_atoms/logP/index)
  - `datasetStats`: statistics payload for display
- React rendering structures:
  - tab selection state (`selectedTab`)
  - Bohr model rendering iterates over `element_bohr_models` map

## Dataset(s)
### Periodic table + Bohr assets (frontend CSV)
- `frontend/public/data/PeriodicTableCSV.csv`
  - Used for element composition → Bohr “2D/3D model” lookup
  - Columns include (among others): `symbol`, `electron_configuration`, `electronegativity_pauling`,
    `bohr_model_image`, `bohr_model_3d`, and discovery/description metadata.

### Bioactivity dataset (backend CSV)
- `backend/data/SMILES_Big_Data_Set.csv` (present in this repo)
  - Used by `/chemistry/dataset-search`, `/dataset-stats`, `/dataset-sample`
  - Required columns referenced by code:
    - `SMILES`, `pIC50`, `num_atoms`, `logP`

### Local chemical-name resolver (optional SDF)
- `backend/data/chembl_sample.sdf` is referenced by the backend loader (`chemistry_core.py`) as `SDF_FILE`,
  - but it is not present in `backend/data/` in the current repo snapshot.
  - If added, it is intended to power local “name/formula → SMILES” resolution before PubChem calls.

### External dataset (PubChem)
- PubChem REST API:
  - resolves common compound names/formulas to SMILES
  - searches by name to return candidate compounds and their SMILES
