# Project Final Summary (Tech Overview)

## What this project is
`retrosynthesis-planner-main` is a multi-page “AI chemistry” web application that combines:
- a **SMILES Converter Studio** (chemical identifiers + descriptors + SMARTS-based reactivity + element Bohr visuals)
- an **Electrolysis Stimulator** (theoretical yield via Faraday’s laws)
- an **Interactive Periodic Table** (CSV-backed element browser + trends)
- a **RetroSynthesis Planner** (neural-guided retrosynthesis search using Retro*)

Each feature is exposed as its own Next.js route (`/converter`, `/electrolysis`, `/periodic`, `/retrosynthesis`) and shares a common UX approach: clear inputs, computed outputs, and educational “behind the formula” explanations.

## Tech stack (overall)
- Frontend: Next.js (React) + TypeScript + Tailwind CSS + shadcn/ui
- Backend services:
  - FastAPI for chemistry endpoints (SMILES conversion, property calculation, dataset browsing/search)
  - FastAPI endpoints for retrosynthesis planning via Retro*
- Chemistry & ML dependencies:
  - RDKit (molecular parsing + descriptors + 2D depiction)
  - PyTorch + Retro* (neural-guided retrosynthesis planning)
  - rdchiral + RDKit tooling inside Retro* for reaction-template application
- External API:
  - PubChem REST for name/formula resolution and compound lookup
- Data:
  - CSV files for:
    - periodic table element data + asset URLs
    - SMILES dataset for property-related browsing/search
  - Retro* model assets under `backend/retro_star/`

## Core logic concepts used across the project
- REST-style separation of concerns:
  - frontend collects inputs and renders results
  - backend performs computational chemistry / ML planning
- “Fallback resolution” patterns:
  - converter tries local datasets first, then PubChem
- Caching of expensive data/model initialization:
  - periodic table + molecule caches in the chemistry backend
  - Retro* planner singleton initialization
- Derived computations presented as educational explanations:
  - Lipinski rule checks
  - SMARTS substructure group detection
  - Faraday’s law dimensional reasoning
  - periodic trend normalization for charting

## Algorithmic highlights (by module)
- SMILES Converter:
  - SMARTS substructure matching to detect functional groups
  - heuristic reactivity tiering from detected functional groups
  - Lipinski’s Rule of Five as a threshold-based classifier
  - pandas-based dataset search using pIC50 range filtering and SMILES substring matching
- Electrolysis Stimulator:
  - Faraday’s law for mapping `I × t × efficiency` → charge → moles → theoretical mass yield
- Interactive Periodic Table:
  - mapping `period/group` into a grid representation + NaN-aware trend normalization
  - category filtering over CSV-derived element objects
- RetroSynthesis Planner:
  - Retro* neural-guided A* search (policy proposal + optional value function)
  - route serialization parsing into UI-friendly step objects

## Main datasets / model assets
- `backend/data/SMILES_Big_Data_Set.csv` (bioactivity dataset used by the converter’s dataset browser/search)
- `frontend/public/data/PeriodicTableCSV.csv` (element properties + image/model URLs for the periodic explorer and Bohr visuals)
- Retro* assets (required for real planning):
  - `backend/retro_star/retro_star/dataset/origin_dict.csv`
  - `backend/retro_star/retro_star/one_step_model/template_rules_1.dat`
  - `backend/retro_star/retro_star/one_step_model/saved_rollout_state_1_2048.ckpt`
  - `backend/retro_star/retro_star/saved_models/best_epoch_final_4.pt`
- PubChem REST API (external dataset used by the converter when local resolution fails)

## Data structures (conceptual summary)
- Backend:
  - dict caches for fast lookup (periodic table, molecule maps)
  - pandas DataFrames for dataset search/statistics
  - typed response objects for stable UI rendering
- Frontend:
  - arrays of typed objects (`Element[]`, `Substance[]`, `ReactionStep[]`)
  - matrix/grid representations for the periodic table (`(Element | null)[][]`)
